# 智链 AI 网关 - 数据库设计

## 一、设计原则

- 基于 **SQLite + Prisma ORM**，与 CoolStartup 脚手架保持一致
- 开启 **WAL 模式**，提升并发写入性能
- 审计日志高频写入通过 **进程内内存缓冲队列** 异步批量落库，无需外部中间件
- 熔断状态、配额计数、限流滑动窗口等运行时状态使用 **进程内 Map** 管理，启动时从 DB 恢复
- 继承脚手架现有的 `User`、`Session`、`SystemConfig`、`LoginRecord` 模型并扩展
- **单实例架构**，零外部依赖（无 Redis），Docker 单容器即可运行

## 四、内存数据结构设计

> **说明**：系统采用单实例架构，所有运行时状态使用进程内 `Map` 管理，无 Redis 依赖。服务启动时从 SQLite 恢复持久化状态。

### 4.1 配额计数器

```
Map Key:  quota:{type}:{refId}:{period}
  type:   token | user | provider
  refId:  对应的 ID
  period: 如 2026-04（月度）

示例：
  quota:token:t456:2026-04      → 令牌月度 Token 用量
  quota:request:t456:2026-04    → 令牌月度请求数
  quota:user:u123:2026-04       → 用户月度 Token 用量
  quota:provider:p789:2026-04   → 提供商月度 Token 用量

操作：
  - 读取：Map.get(key) || 0
  - 扣减：Map.set(key, (current || 0) + delta)  （同步操作，无 await，线程安全）
  - 启动恢复：从 QuotaSnapshot 表读取当期快照，补充从 AuditLog 聚合未快照的用量
  - 定时持久化：每 60 秒将内存计数 upsert 到 QuotaSnapshot 表
  - 周期重置：每天 0:01 检测新周期，清空过期 key，新周期从 0 开始
```

### 4.2 限流计数器（滑动窗口）

```
Map Key:  ratelimit:{type}:{refId}:{window}
  type:   token | provider
  refId:  对应的 ID
  window: rpm | tpm

Value:   number[]（请求时间戳数组）

操作：
  - 清除过期：array.filter(ts => ts > Date.now() - 60000)  （保留最近 60s）
  - 计数：array.length
  - 添加：array.push(Date.now())
  - 判断：清除过期后，若 array.length >= limit 则拒绝（429）
```

### 4.3 熔断器状态

```
Map Key:  circuit:{providerId}

Value: {
  state: "closed" | "open" | "half_open",
  successWindow: number[],   // 滑动窗口：请求成功时间戳
  failWindow: number[],      // 滑动窗口：请求失败时间戳
  since: number              // 状态变更时间戳（用于冷却期判断）
}

操作：
  - 记录成功：successWindow.push(Date.now()) + trimWindow()
  - 记录失败：failWindow.push(Date.now()) + trimWindow() + 计算错误率判定是否熔断
  - 判断可用：state === "closed" || (state === "open" && 冷却期已过 → 半开)
  - 状态转换：closed → open（错误率超阈值）、open → half_open（冷却期后）、half_open → closed（探测成功）
```

### 4.4 审计日志异步写入缓冲

```
数据结构：AuditLogEntry[]（数组缓冲区）

生产：请求完成后将日志对象 push 到数组（O(1)）
消费：每 2 秒或缓冲区满 100 条时，splice 取出批量 INSERT 到 SQLite

异常处理：
  - 落库失败时回塞缓冲区头部，下次重试
  - 缓冲区积压超过 10000 条时触发告警
  - 优雅关闭时（SIGTERM）执行 flush 将剩余日志全部落库
```

### 4.5 启动恢复流程

```
服务启动时（instrumentation.ts register hook）：

1. 配额恢复：
   - 从 QuotaSnapshot 读取当期快照 → 回填 Map
   - 从 AuditLog 按 period 聚合 SUM(totalTokens) → 补充未快照的用量
   
2. 熔断器初始化：
   - 所有提供商初始状态为 closed（不信任上次运行时状态）
   
3. 审计日志消费者：
   - 启动定时 flush interval（2s/100条）
   
4. 健康探测：
   - 启动定时探测 interval（60s）
   
5. 配额快照定时器：
   - 启动定时持久化 interval（60s）
```

## 五、SQLite 性能优化策略

### 5.1 WAL 模式

在 `src/lib/prisma.ts` 启动时执行：

```typescript
await prisma.$executeRawUnsafe('PRAGMA journal_mode=WAL');
await prisma.$executeRawUnsafe('PRAGMA synchronous=NORMAL');
await prisma.$executeRawUnsafe('PRAGMA cache_size=-64000'); // 64MB
await prisma.$executeRawUnsafe('PRAGMA busy_timeout=5000');
```

### 5.2 审计日志异步写入

```
请求 → 生成审计日志 → push 到内存缓冲区（数组）→ 立即返回响应
                                       │
                               定时消费者（每2s/每100条）
                                       │
                               批量 INSERT 到 SQLite
```

### 5.3 日志归档策略

- 通过 `SystemConfig` 配置保留天数（默认 30 天）
- 定时任务每天凌晨执行归档：
  1. 超过保留期的日志导出为 JSON 文件（压缩存储）
  2. 归档完成后删除已归档的数据库记录
  3. 归档文件存储在 `data/archives/` 目录

### 5.4 数据库备份

- 继承脚手架的自动备份机制，保留最近 5 个备份
- 备份触发：版本升级时、每日定时
- 备份目录：`data/backups/`
